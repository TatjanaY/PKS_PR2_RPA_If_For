# Вариант 8

## Финансовый трекер расходов

---

## Контекст задачи

Компания запускает корпоративное мобильное приложение для финансового трекинга сотрудников. Приложение фиксирует расходы по категориям, бюджетные лимиты и сроки проведения финансовых операций.

Менеджмент сталкивается с задачами:

* часть сотрудников превышает выделенные бюджеты по категориям расходов;
* отдельные расходы просрочены, что влияет на корректность отчётности;
* необходимо выявлять категории с высоким финансовым риском и составлять управленческий отчёт;
* формировать историю расходов для анализа динамики потребления средств.

**Цель кейса:** реализовать RPA-процесс, который автоматически анализирует финансовые данные сотрудников, рассчитывает отклонения, классифицирует расходы по уровню риска, формирует итоговый Excel-отчёт и выводит уведомление в модальном окне.

---

# Задание 1. Анализ расходов за январь (базовый уровень)

**Выполняется в Puzzle RPA и Excel.**

**История:** Вы — финансовый аналитик. Необходимо оценить выполнение бюджета сотрудников за первый месяц года, выявить перерасходы и просроченные операции, а затем подготовить сводку для управленческого отчёта.

---

## 1. Подготовка данных (Excel)

Создать файл `Finance_Tracker_January.xlsx` с листом `Expenses_Jan`.

**Структура таблицы:**

| Поле            | Тип     | Смысл                                                 |
| --------------- | ------- | ----------------------------------------------------- |
| TransactionID   | string  | Уникальный идентификатор расхода                      |
| EmployeeName    | string  | ФИО сотрудника                                        |
| Category        | string  | Категория расхода (Travel, Meals, Supplies, Software) |
| Budget          | decimal | Выделенный бюджет на категорию, тыс. $                |
| Spent           | decimal | Фактические расходы в тыс. $                          |
| TransactionDate | date    | Дата совершения расхода                               |
| DueDate         | date    | Срок утверждения или списания расхода                 |

**Методический комментарий:** таблица отражает реальные расходы сотрудников. Данные генерируются через DeepSeek с разнообразием сумм, дат и категорий.

---

## 2. Инициализация переменных и счётчиков (Puzzle RPA)

| Счётчик                  | Начальное значение | Смысл                                              |
| ------------------------ | ------------------ | -------------------------------------------------- |
| total_transactions       | 0                  | Общее количество операций                          |
| over_budget_transactions | 0                  | Количество расходов, превысивших бюджет            |
| expired_transactions     | 0                  | Количество просроченных расходов                   |
| high_risk_categories     | 0                  | Количество категорий расходов с перерасходом > 20% |
| total_overbudget_amount  | 0                  | Суммарный перерасход по всем операциям, тыс. $     |
| total_transactions_value | 0                  | Суммарная стоимость всех операций, тыс. $          |

---

## 3. Преобразование данных (Puzzle RPA, на каждой строке таблицы)

* `budget, spent → decimal`
* `transactiondate, duedate → date`
* `employeename → удалить пробелы`
* `category → верхний регистр`

**Методический комментарий:** корректное приведение типов позволяет точно рассчитывать отклонения, сравнивать даты и суммировать финансовые показатели.

---

## 4. Расчёты (Puzzle RPA, на каждой строке таблицы)

* `deviation = spent - budget`
  *абсолютное отклонение расходов в тыс. $*

* `deviation_percent = (spent - budget) / budget * 100`
  *процентное отклонение бюджета по категории*

* `remaining_days = duedate - currentdate`
  *количество дней до завершения операции; отрицательное значение → просрочка*

**Методический комментарий:** вычисления позволяют выявить перерасходы и просрочки на уровне каждой операции.

---

## 5. Логика классификации расходов (Puzzle RPA, на каждой строке таблицы)

**Простые условия:**

* если `spent > budget` → `over_budget_transactions += 1`, `total_overbudget_amount += deviation`
* если `duedate < currentdate` → `expired_transactions += 1`

**Составные условия:**

* если `spent > budget AND deviation_percent > 20` → `high_risk_categories += 1`
* если `deviation_percent > 10 OR duedate < currentdate` → операция попадает в зону риска

**Вложенные условия:**

* если `deviation_percent > 20` → категория считается критической
* иначе → категория управляемая

**Методический комментарий:** простые условия фиксируют перерасход и просрочку, составные — выявляют рискованные операции, вложенные — определяют итоговую категорию риска. Выполняется для каждой строки таблицы.

---

## 6. Формирование аналитической таблицы (Excel)

Файл `Finance_Tracker_Result.xlsx`, лист `Analysis_Jan` с полями:

| TransactionID | EmployeeName | Category | Deviation | DeviationPercent | RemainingDays | RiskCategory |

---

## 7. Итоговое уведомление (Puzzle RPA, модальное окно)

```
f"""Анализ расходов за январь завершён.

Всего операций: {total_transactions}
С перерасходом: {over_budget_transactions}
Просроченных: {expired_transactions}
Категорий с критическим превышением бюджета: {high_risk_categories}
Суммарный перерасход: {total_overbudget_amount} тыс. $

Отчёт сохранён в Finance_Tracker_Result.xlsx (лист Analysis_Jan).
"""
```

# Задание 2. Прогнозная оценка расходов за февраль (расширенный уровень)

**Выполняется в Puzzle RPA и Excel.**

**История:** Во втором месяце планируется более активная финансовая деятельность сотрудников. Необходимо оценить потенциальные перерасходы, прогнозировать риск по каждой категории и сотруднику, используя цикл по строкам таблицы.

---

## 1. Подготовка данных (Excel)

Файл `Finance_Tracker_February.xlsx`, лист `Expenses_Feb` с аналогичной структурой, но добавлены новые поля:

| Поле            | Тип     | Смысл                                   |
| --------------- | ------- | --------------------------------------- |
| TransactionID   | string  | Уникальный идентификатор операции       |
| EmployeeName    | string  | ФИО сотрудника                          |
| Category        | string  | Категория расхода                       |
| Budget          | decimal | Бюджет на категорию                     |
| Spent           | decimal | Фактические расходы                     |
| TransactionDate | date    | Дата расхода                            |
| DueDate         | date    | Срок утверждения                        |
| ExpectedSpend   | decimal | Ожидаемая сумма расхода на конец месяца |
| Notes           | string  | Комментарии сотрудника                  |

**Методический комментарий:** новые поля позволяют строить прогнозные оценки и историю расходов.

---

## 2. Инициализация переменных и счётчиков (Puzzle RPA)

| Счётчик                  | Начальное значение | Смысл                                                  |
| ------------------------ | ------------------ | ------------------------------------------------------ |
| total_transactions       | 0                  | Общее количество операций                              |
| over_budget_transactions | 0                  | Количество перерасходов                                |
| expired_transactions     | 0                  | Количество просрочек                                   |
| high_risk_categories     | 0                  | Количество категорий с критическим превышением бюджета |
| total_overbudget_amount  | 0                  | Суммарный перерасход по операциям                      |
| total_transactions_value | 0                  | Суммарная стоимость всех операций                      |
| total_forecast_risk      | 0                  | Суммарный прогнозный риск по портфелю расходов         |

---

## 3. Цикл по строкам таблицы (Puzzle RPA)

На каждой итерации:

1. Привести типы данных:

* `budget, spent, expectedspend → decimal`
* `transactiondate, duedate → date`
* `employeename → удалить пробелы`
* `category → верхний регистр`

2. Вычислить показатели:

* `deviation = spent - budget`
* `deviation_percent = (spent - budget) / budget * 100`
* `remaining_days = duedate - currentdate`
* `forecast_deviation = (expectedspend - budget) / budget * 100`

**Методический комментарий:** прогнозное отклонение показывает потенциальный перерасход до конца месяца.

---

## 4. Многоуровневая логика прогнозного риска (Puzzle RPA, на каждой строке)

* если `deviation_percent > 20 OR forecast_deviation > 25` → `risk_category = "Critical"`, `high_risk_categories += 1`

* если `deviation_percent > 10 OR forecast_deviation > 15` → `risk_category = "High Risk"`

* если `deviation_percent > 5 OR forecast_deviation > 10` → `risk_category = "Moderate Risk"`

* иначе → `risk_category = "Stable"`

* если `duedate < currentdate` → `expired_transactions += 1`

* если `spent > budget` → `over_budget_transactions += 1`, `total_overbudget_amount += deviation`

**Методический комментарий:** простые условия фиксируют перерасход и просрочку, составные — прогнозируют риск по динамике расходов, вложенные — определяют итоговую категорию риска.

---

## 5. Формирование результирующего Excel (Excel)

Файл `Finance_Tracker_Result.xlsx`, лист `Analysis_Feb` с полями:

| TransactionID | EmployeeName | Category | Deviation | DeviationPercent | ForecastDeviation | RemainingDays | RiskCategory |

---

```
Просроченных: {expired_transactions}
Категорий с критическим риском: {high_risk_categories}
Суммарный перерасход: {total_overbudget_amount} тыс. $
Суммарный прогнозный риск: {total_forecast_risk}%

Отчёт сохранён в Finance_Tracker_Result.xlsx (лист Analysis_Feb).
"""
```
