# Вариант 2

**Тематика:** Мониторинг и анализ активности пользователей мобильного фитнес-приложения

---

## Контекст задачи

Компания-разработчик мобильного приложения для фитнес-трекеров ведёт более 10 000 активных пользователей, с различной частотой тренировок, типами активности и целями по здоровью.

Руководство сталкивается с типичными проблемами:

* часть пользователей не достигает запланированных целей активности;
* наблюдаются просрочки по ежедневным и еженедельным тренировкам;
* нет единой классификации уровня вовлечённости пользователей и потенциального риска отказа от приложения;
* требуется агрегированная аналитика для формирования рекомендаций и уведомлений пользователю.

**Цель кейса:** реализовать RPA-процесс, который анализирует портфель пользователей, рассчитывает ключевые показатели активности и вовлечённости, классифицирует пользователей по уровню риска неисполнения целей, формирует отчёт в Excel и выводит уведомление в модальном окне.

---

# Задание 1. Анализ активности пользователей Q1 (базовый уровень)

**Выполняется в Puzzle RPA и Excel.**

**История:** В первом квартале приложение фиксировало активность пользователей. Необходимо оценить, какие пользователи выполняют свои цели, какие отклоняются от плана, выявить просрочки и сформировать сводный отчёт.

---

## 1. Подготовка данных (Excel)

Создать файл `FitnessApp_Users_Q1.xlsx` с листом `Users_Q1`.

**Структура таблицы:**

| Поле             | Тип     | Смысл                                         |
| ---------------- | ------- | --------------------------------------------- |
| UserID           | string  | Уникальный идентификатор пользователя         |
| Status           | string  | Статус активности: Active, Inactive, Trial    |
| DailyGoal        | decimal | Планируемая ежедневная активность (минут)     |
| ActualActivity   | decimal | Фактическая активность пользователя (минут)   |
| RegistrationDate | date    | Дата регистрации в приложении                 |
| LastActivity     | date    | Дата последней активности                     |
| AgeGroup         | string  | Возрастная группа (18-25, 26-35, 36-50, 50+)  |
| ActivityType     | string  | Тип активности: Cardio, Strength, Yoga, Mixed |

**Методический комментарий:** таблица отражает исходные данные пользователей. Значения генерируются через DeepSeek для разнообразия активности, дат и возрастных групп.

---

## 2. Инициализация переменных и счётчиков (Puzzle RPA)

| Счётчик               | Начальное значение | Смысл                                                            |
| --------------------- | ------------------ | ---------------------------------------------------------------- |
| total_users           | 0                  | Общее количество пользователей                                   |
| active_users          | 0                  | Количество активных пользователей                                |
| inactive_users        | 0                  | Количество неактивных пользователей                              |
| trial_users           | 0                  | Количество пользователей на пробной версии                       |
| underperform_users    | 0                  | Пользователи с активностью ниже DailyGoal                        |
| expired_users         | 0                  | Пользователи без активности за последние 7 дней                  |
| risk_users            | 0                  | Пользователи с высоким риском отказа                             |
| critical_users        | 0                  | Пользователи с критическим отклонением по цели                   |
| totalUnderperformLoad | 0                  | Суммарное недовыполнение целей активными пользователями (минуты) |

---

## 3. Преобразование данных (Puzzle RPA, на каждой строке таблицы)

* `DailyGoal, ActualActivity → decimal`
* `RegistrationDate, LastActivity → date`
* `ActivityType → верхний регистр`
* `AgeGroup → удалить пробелы`

**Методический комментарий:** корректные типы обеспечивают точность вычислений и корректное сравнение чисел и дат.

---

## 4. Расчёты (Puzzle RPA, на каждой строке таблицы)

* `deviation = ActualActivity - DailyGoal`
  *разница между фактической и планируемой активностью*

* `deviation_percent = (DailyGoal - ActualActivity) / DailyGoal * 100`
  *процент недовыполнения цели*

* `inactive_days = CurrentDate - LastActivity`
  *количество дней без активности; больше 7 → просрочка*

**Методический комментарий:** расчёты помогают определить пользователей с недовыполнением целей и просроченной активностью.

---

## 5. Логика классификации пользователей (Puzzle RPA, на каждой строке таблицы)

**Простые условия:**

* если `ActualActivity < DailyGoal` → `underperform_users += 1`
* если `inactive_days > 7` → `expired_users += 1`

**Составные условия:**

* если `ActualActivity < DailyGoal AND Status = "Active"` → `totalUnderperformLoad += deviation`
  *финансовая/поведенческая нагрузка активных пользователей*

* если `deviation_percent > 20 OR inactive_days > 7` → `risk_users += 1`

**Вложенные условия:**

* если `Status = "Active"`:

  * если `deviation_percent > 40` → `critical_users += 1`
  * иначе → пользователь управляемый
* если `Status = "Inactive"` → `inactive_users += 1`
* если `Status = "Trial"` → `trial_users += 1`

**Методический комментарий:** простые условия фиксируют факты (недовыполнение целей, просрочку), составные — суммарное недовыполнение и зоны риска, вложенные — итоговую категорию пользователя.

---

## 6. Формирование аналитической таблицы (Excel)

Файл `FitnessApp_Users_Result.xlsx`, лист `Analysis_Q1` с полями:

| UserID | Deviation | DeviationPercent | InactiveDays | RiskCategory | IsExpired |

---

## 7. Итоговое уведомление (Puzzle RPA, модальное окно)

```
f"""Анализ портфеля пользователей Q1 завершён.

Всего пользователей: {total_users}
Активных: {active_users}
Неактивных: {inactive_users}
На пробной версии: {trial_users}

С недовыполнением целей: {underperform_users}
Просроченных: {expired_users}
Рисковых: {risk_users}
Критических: {critical_users}
Суммарное недовыполнение целей активных пользователей: {totalUnderperformLoad} минут

Отчёт сохранён в FitnessApp_Users_Result.xlsx (лист Analysis_Q1).
"""
```

# Задание 2. Расширенный анализ активности Q2 с прогнозной оценкой риска

**Выполняется в Puzzle RPA и Excel.**

**История:** Во втором квартале количество пользователей увеличилось, появляются новые паттерны активности. Необходимо оценить прогнозное недовыполнение целей на оставшуюся часть квартала с использованием цикла по строкам таблицы и вычислением прогнозных показателей.

---

## 1. Подготовка данных (Excel)

Файл `FitnessApp_Users_Q2.xlsx`, лист `Users_Q2` с той же структурой, что и Q1. Данные генерируются через DeepSeek.

---

## 2. Инициализация переменных и счётчиков (Puzzle RPA)

| Счётчик               | Начальное значение | Смысл                                                            |
| --------------------- | ------------------ | ---------------------------------------------------------------- |
| total_users           | 0                  | Общее количество пользователей                                   |
| active_users          | 0                  | Количество активных пользователей                                |
| inactive_users        | 0                  | Количество неактивных пользователей                              |
| trial_users           | 0                  | Количество пользователей на пробной версии                       |
| underperform_users    | 0                  | Пользователи с активностью ниже DailyGoal                        |
| expired_users         | 0                  | Пользователи без активности более 7 дней                         |
| risk_users            | 0                  | Пользователи с высоким риском неисполнения цели                  |
| critical_users        | 0                  | Пользователи с критическим недовыполнением                       |
| totalUnderperformLoad | 0                  | Суммарное недовыполнение целей активными пользователями (минуты) |
| totalForecastRisk     | 0                  | Суммарный прогнозный риск по портфелю пользователей              |

---

## 3. Цикл по строкам таблицы (Puzzle RPA)

На каждой итерации:

1. Привести типы и формат данных:

* `DailyGoal, ActualActivity → decimal`
* `RegistrationDate, LastActivity → date`
* `ActivityType → верхний регистр`
* `AgeGroup → удалить пробелы`

2. Вычислить:

* `deviation = ActualActivity - DailyGoal`
* `deviation_percent = (DailyGoal - ActualActivity) / DailyGoal * 100`
* `inactive_days = CurrentDate - LastActivity`
* `days_since_registration = CurrentDate - RegistrationDate`
* `forecast_deviation = deviation_percent * (30 - inactive_days)/30`

**Методический комментарий:** прогнозное отклонение отражает потенциальное недовыполнение цели до конца месяца.

---

## 4. Многоуровневая логика прогнозного риска (Puzzle RPA, на каждой строке)

* если `Status = "Active"`:

  * если `inactive_days > 7` (пользователь не активен):

    * если `deviation_percent > 40` → `risk_category = "Critical"`, `critical_users += 1`
    * иначе → `risk_category = "Delayed"`
  * иначе:

    * если `forecast_deviation > 50` → `risk_category = "High Risk"`, `critical_users += 1`
    * если `forecast_deviation > 30` → `risk_category = "Moderate Risk"`, `risk_users += 1`
    * иначе → `risk_category = "On Track"`
* если `Status = "Inactive"` → `risk_category = "Dormant"`
* если `Status = "Trial"` → `risk_category = "Monitoring"`

**Методический комментарий:** выполняется для каждой строки, классифицирует пользователей по совокупности фактической активности и прогнозного недовыполнения цели.

---

## 5. Формирование результирующего Excel (Excel)

Файл `FitnessApp_Users_Result.xlsx`, лист `Analysis_Q2` с полями:

| UserID | DeviationPercent | InactiveDays | ForecastDeviation | RiskCategory | FinalStatus |
