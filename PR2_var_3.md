# Вариант 3

**Тематика:** Анализ продаж и взаимодействия с клиентами в корпоративной CRM-системе

---

## Контекст задачи

Корпоративная компания использует CRM-систему для управления более чем 5 000 клиентов и потенциальных сделок.

Руководство сталкивается с типичными проблемами:

* часть клиентов неактивны или не реагируют на коммерческие предложения;
* сделки могут задерживаться по срокам согласования и оплаты;
* нет единой классификации риска упущенных сделок;
* требуется агрегированная аналитика по клиентам и сделкам для формирования отчётов и планирования действий менеджеров.

**Цель кейса:** реализовать RPA-процесс, который анализирует базу клиентов и сделок, рассчитывает показатели активности и финансового статуса сделок, классифицирует клиентов и сделки по уровню риска, формирует отчёт в Excel и выводит уведомление в модальном окне.

---

# Задание 1. Анализ портфеля сделок CRM Q1 (базовый уровень)

**Выполняется в Puzzle RPA и Excel.**

**История:** В первом квартале менеджеры компании обрабатывали сделки с клиентами. Необходимо определить, какие сделки идут с отклонениями по срокам и бюджету, выявить клиентов с низкой активностью, и подготовить сводный отчёт для руководства.

---

## 1. Подготовка данных (Excel)

Создать файл `CRM_Deals_Q1.xlsx` с листом `Deals_Q1`.

**Структура таблицы:**

| Поле         | Тип     | Смысл                                         |
| ------------ | ------- | --------------------------------------------- |
| DealID       | string  | Уникальный идентификатор сделки               |
| ClientStatus | string  | Статус клиента: Active, Inactive, Prospective |
| DealValue    | decimal | Сумма сделки ($)                              |
| AmountPaid   | decimal | Фактическая оплата по сделке ($)              |
| StartDate    | date    | Дата начала сделки                            |
| DueDate      | date    | Дата завершения сделки                        |
| ManagerName  | string  | Имя менеджера, ответственного за сделку       |
| DealType     | string  | Тип сделки: NewBusiness, Renewal, Upsell      |

**Методический комментарий:** таблица отражает исходные данные по сделкам и клиентам. Значения генерируются через DeepSeek для разнообразия сумм, дат и типов сделок.

---

## 2. Инициализация переменных и счётчиков (Puzzle RPA)

| Счётчик               | Начальное значение | Смысл                                               |
| --------------------- | ------------------ | --------------------------------------------------- |
| total_deals           | 0                  | Общее количество сделок                             |
| active_clients        | 0                  | Количество активных клиентов                        |
| inactive_clients      | 0                  | Количество неактивных клиентов                      |
| prospective_clients   | 0                  | Количество потенциальных клиентов                   |
| overpaid_deals        | 0                  | Сделки, где AmountPaid > DealValue                  |
| delayed_deals         | 0                  | Сделки с просроченной датой DueDate                 |
| risk_deals            | 0                  | Сделки с высоким риском недоплаты или просрочки     |
| critical_deals        | 0                  | Сделки с критическим отклонением по оплате и срокам |
| totalOutstandingValue | 0                  | Суммарная недоплата по активным сделкам ($)         |

---

## 3. Преобразование данных (Puzzle RPA, на каждой строке таблицы)

* `DealValue, AmountPaid → decimal`
* `StartDate, DueDate → date`
* `ManagerName → удалить пробелы`
* `DealType → верхний регистр`

**Методический комментарий:** корректные типы данных обеспечивают точность вычислений и корректное сравнение чисел и дат.

---

## 4. Расчёты (Puzzle RPA, на каждой строке таблицы)

* `deviation = DealValue - AmountPaid`
  *разница между суммой сделки и фактической оплатой*

* `deviation_percent = (DealValue - AmountPaid)/DealValue * 100`
  *процент недоплаты по сделке*

* `remaining_days = DueDate - CurrentDate`
  *количество дней до окончания сделки; отрицательное значение → просрочка*

**Методический комментарий:** эти расчёты позволяют определить сделки с недоплатой и задержкой по срокам.

---

## 5. Логика классификации сделок (Puzzle RPA, на каждой строке таблицы)

**Простые условия:**

* если `AmountPaid > DealValue` → `overpaid_deals += 1`
* если `DueDate < CurrentDate` → `delayed_deals += 1`

**Составные условия:**

* если `AmountPaid < DealValue AND ClientStatus = "Active"` → `totalOutstandingValue += deviation`
  *финансовая нагрузка по активным сделкам*

* если `deviation_percent > 20 OR DueDate < CurrentDate` → `risk_deals += 1`

**Вложенные условия:**

* если `ClientStatus = "Active"`:

  * если `deviation_percent > 40` → `critical_deals += 1`
  * иначе → сделка управляемая
* если `ClientStatus = "Inactive"` → `inactive_clients += 1`
* если `ClientStatus = "Prospective"` → `prospective_clients += 1`

**Методический комментарий:** простые условия фиксируют факты (переплата, просрочка), составные — суммарную финансовую нагрузку и зоны риска, вложенные — итоговую категорию сделки.

---

## 6. Формирование аналитической таблицы (Excel)

Файл `CRM_Deals_Result.xlsx`, лист `Analysis_Q1` с полями:

| DealID | Deviation | DeviationPercent | RemainingDays | RiskCategory | IsDelayed |

---

## 7. Итоговое уведомление (Puzzle RPA, модальное окно)

```
f"""Анализ портфеля сделок CRM Q1 завершён.

Всего сделок: {total_deals}
Активных клиентов: {active_clients}
Неактивных клиентов: {inactive_clients}
Потенциальных клиентов: {prospective_clients}

С переплатой: {overpaid_deals}
Просроченных: {delayed_deals}
Рисковых: {risk_deals}
Критических: {critical_deals}
Суммарная недоплата по активным сделкам: {totalOutstandingValue} $

Отчёт сохранён в CRM_Deals_Result.xlsx (лист Analysis_Q1).
"""
```

# Задание 2. Расширенный анализ CRM Q2 с прогнозной оценкой риска

**Выполняется в Puzzle RPA и Excel.**

**История:** Во втором квартале количество сделок и клиентов увеличилось, появляются новые паттерны оплаты и задержек. Необходимо спрогнозировать риск недоплаты и задержек по каждой сделке с использованием цикла по строкам таблицы.

---

## 1. Подготовка данных (Excel)

Файл `CRM_Deals_Q2.xlsx`, лист `Deals_Q2` с той же структурой, что и Q1. Данные генерируются через DeepSeek.

---

## 2. Инициализация переменных и счётчиков (Puzzle RPA)

| Счётчик               | Начальное значение | Смысл                                               |
| --------------------- | ------------------ | --------------------------------------------------- |
| total_deals           | 0                  | Общее количество сделок                             |
| active_clients        | 0                  | Количество активных клиентов                        |
| inactive_clients      | 0                  | Количество неактивных клиентов                      |
| prospective_clients   | 0                  | Количество потенциальных клиентов                   |
| overpaid_deals        | 0                  | Сделки с переплатой                                 |
| delayed_deals         | 0                  | Сделки с просрочкой                                 |
| risk_deals            | 0                  | Сделки с высоким риском недоплаты или задержки      |
| critical_deals        | 0                  | Сделки с критическим отклонением по оплате и срокам |
| totalOutstandingValue | 0                  | Суммарная недоплата по активным сделкам ($)         |
| totalForecastRisk     | 0                  | Суммарный прогнозный риск по портфелю сделок        |

---

## 3. Цикл по строкам таблицы (Puzzle RPA)

На каждой итерации:

1. Привести типы и формат данных:

* `DealValue, AmountPaid → decimal`
* `StartDate, DueDate → date`
* `ManagerName → удалить пробелы`
* `DealType → верхний регистр`

2. Вычислить:

* `deviation = DealValue - AmountPaid`
* `deviation_percent = (DealValue - AmountPaid)/DealValue * 100`
* `remaining_days = DueDate - CurrentDate`
* `total_duration_days = DueDate - StartDate`
* `forecast_deviation = deviation_percent * (remaining_days / total_duration_days)`

**Методический комментарий:** прогнозное отклонение отражает потенциальный риск недоплаты и просрочки до конца сделки.

---

## 4. Многоуровневая логика прогнозного риска (Puzzle RPA, на каждой строке)

* если `ClientStatus = "Active"`:

  * если `remaining_days < 0` (сделка просрочена):

    * если `deviation_percent > 40` → `risk_category = "Critical"`, `critical_deals += 1`
    * иначе → `risk_category = "Delayed"`
  * иначе:

    * если `forecast_deviation > 50` → `risk_category = "High Risk"`, `critical_deals += 1`
    * если `forecast_deviation > 30` → `risk_category = "Moderate Risk"`, `risk_deals += 1`
    * иначе → `risk_category = "On Track"`
* если `ClientStatus = "Inactive"` → `risk_category = "Dormant"`
* если `ClientStatus = "Prospective"` → `risk_category = "Monitoring"`

**Методический комментарий:** выполняется для каждой строки, классифицирует сделки по совокупности финансового и временного риска.

---

## 5. Формирование результирующего Excel (Excel)

Файл `CRM_Deals_Result.xlsx`, лист `Analysis_Q2` с полями:

| DealID | DeviationPercent | RemainingDays | ForecastDeviation | RiskCategory | FinalStatus |

---

```
Потенциальных клиентов: {prospective_clients}

Рисковых: {risk_deals}
Просроченных: {delayed_deals}
Суммарная недоплата по активным сделкам: {totalOutstandingValue} $
Суммарный прогнозный риск портфеля: {totalForecastRisk}%

Отчёт сформирован в CRM_Deals_Result.xlsx (лист Analysis_Q2).
"""
```
